# uTools 系统监控插件性能优化设计

日期：2026-06-13

## 目标

在不明显影响用户体验和监控效果的前提下，尽可能降低插件自身的 CPU 和内存占用。

本次设计约束如下：

- 用户可以自定义刷新策略
- 默认体验为“平衡”档，目标刷新感知约为 3-5 秒
- 切换页面后，页面实例不销毁
- 切走的页面不继续更新动态数据
- 窗口失焦、最小化或不可见时，自动进一步降频

## 当前问题

结合现有实现，主要性能成本来自以下几个方面：

- `src/composables/useHardwareData.ts` 采用全局单例 store 和统一动态轮询，多个页面共享同一套更新链路
- `src/App.vue` 使用 `v-show` 保留多个大页面组件，页面实例、局部 `watch`、`computed` 和模板依赖长期常驻
- 当前轮询虽然已经按指标分层，但仍然主要按“全局 store”思路运行，而不是按“当前页面需求”运行
- GPU、主板 telemetry、Windows/macOS 传感器增强兜底链路单位成本较高
- 某些失败或不支持的数据源缺少更激进的负缓存策略，存在重复尝试成本

## 设计结论

采用“两级 store + 统一调度层”的长期方案，而不是继续扩展单一大 store。

整体结构：

- `overview-lite store`
  - 常驻
  - 只保存概览页和跨页面都需要的轻量数据
  - 允许在前台低成本持续刷新
- `page-full stores`
  - 每个详情页一个独立 store
  - 只保存该页面独有的重数据和重派生数据
  - 页面切走后保留实例和上次结果，但停止动态刷新
- `scheduler`
  - 统一决定各类数据何时刷新
  - 决策输入包括：当前活跃页面、窗口可见性、用户刷新档位、数据链路权重、失败负缓存状态

## 备选方案对比

### 方案 A：继续保留单例大 store，仅追加更多条件判断

优点：

- 改动面最小
- 可快速止血

缺点：

- store 职责继续膨胀
- 规则会越来越分散
- 长期维护成本高
- 不利于后续增加更多模式和页面

### 方案 B：两级 store + 调度层

优点：

- 数据职责边界清晰
- 性能控制点集中
- 适合后续增加刷新档位、后台降频、小窗模式和更多传感器能力

缺点：

- 改造量比方案 A 大
- 需要重新整理数据边界和页面依赖

### 方案 C：彻底任务化的采样引擎

优点：

- 理论上扩展性最强

缺点：

- 对当前项目过重
- 短期投入产出比不高

推荐结论：

- 长期采用方案 B
- 落地时分阶段推进，先保住现有体验，再逐步完成结构迁移

## 数据分层

### overview-lite store

只保留“打开即见、适合持续低成本刷新”的数据：

- CPU：总占用、概览温度、概览频率
- 内存：总量、已用、可用、使用率
- GPU：主 GPU 简版数据，只包含占用、温度、功耗或频率中的核心展示字段
- 存储：总体使用率或主盘使用率
- 时间：运行时长、最近同步时间
- 静态摘要：CPU 型号、总内存、主 GPU 名称、系统平台等

设计原则：

- 不为详情页字段买单
- 不保存完整 GPU 列表、完整磁盘列表、主板 telemetry、完整每核 CPU 数据
- 适合默认“平衡”档持续刷新

### page-full stores

#### processorStore

- `cpuLoadData.cpus`
- 更完整的温度、功耗、电压、风扇数据
- 处理器页趋势图所需的历史数据
- 处理器页独有的压力测试状态

#### graphicsStore

- 完整 GPU 列表
- 显存、核心频率、风扇、fallback diagnostics
- 多 GPU 选择逻辑和详情页专属派生数据

#### boardStore

- board telemetry
- BIOS、baseboard、network interface、layout 等主板页拼装数据

#### memoryStore

- memory layout
- 通道、条子、频率、电压、制造信息等详情字段

#### storageStore

- disk layout
- 完整磁盘列表
- 健康状态、SMART 摘要、焦点磁盘详情

## 调度策略

统一由调度层根据以下四个维度叠加决策：

### 1. 用户刷新档位

- `省电`
  - 前台目标感知约 6-10 秒
- `平衡`
  - 默认档
  - 前台目标感知约 3-5 秒
- `实时`
  - 前台目标感知约 1-2 秒

### 2. 窗口状态

- 前台可见：按用户档位正常刷新
- 失焦、最小化或不可见：自动降到前台的 2-4 倍间隔
- 后台状态暂停重链路刷新，优先保留轻链路低频更新

### 3. 页面活跃态

- 当前页面：
  - 允许对应 `page-full store` 动态刷新
- 非当前页面：
  - 保留页面实例
  - 保留上次数据
  - 停止对应 `page-full store` 的动态刷新

### 4. 链路权重

轻链路：

- CPU 总占用
- 概览温度
- 内存信息
- 时间信息
- 主 GPU 简版信息

重链路：

- GPU 全量详情
- board telemetry
- Windows/macOS 传感器增强兜底链路
- SMART/健康探测

规则：

- 轻链路可在前台常驻，后台低频刷新
- 重链路只在对应页面激活且窗口可见时刷新

## 负缓存与失败策略

对高成本且容易失败的数据源增加负缓存：

- 当数据源明确返回“不支持”“无权限”“当前平台不可用”时，进入较长 TTL
- 建议失败 TTL 为 30-60 秒
- 用户切回对应页面或主动手动刷新时，可提前解除负缓存

优先纳入负缓存的链路：

- `getGpuInfo`
- `getBoardTelemetry`
- Windows 硬件传感器增强链路
- macOS 额外传感器兜底链路

设计目标：

- 避免高成本失败源在短周期内被反复尝试
- 将开销集中在真正有价值的刷新上

## 页面与实例策略

本次设计不采用“切页销毁页面”的方案。

页面策略如下：

- 页面组件实例保留
- 页面切走后停止其 `page-full store` 的动态订阅
- 页面切回时直接复用上次状态和结构
- 如有必要，可在重新激活页面时补做一次立即刷新

这样可以兼顾：

- 切页回弹速度
- 页面局部状态保留
- 非当前页不再持续消耗 CPU

## 设置项设计

设置采用“简单档位 + 高级设置”的组合方式。

### 简单档位

提供三个预设：

- `省电`
- `平衡`
- `实时`

默认选中 `平衡`。

### 高级设置

允许高级用户按模块配置刷新策略，例如：

- CPU
- GPU
- 内存
- 存储

高级设置只覆盖对应模块的前台刷新频率；后台降频策略仍由统一调度层接管。

这样可以兼顾：

- 普通用户的低学习成本
- 高级用户对实时性和功耗的自主权

## 迁移顺序

### 第一阶段：建立调度抽象

- 在现有实现基础上引入刷新档位、窗口状态、页面活跃态
- 先把“什么时候更新”从页面中抽离到调度层
- 不立即拆完所有 store，先让当前页和后台行为正确

目标：

- 快速降低无意义刷新
- 保持当前功能稳定

### 第二阶段：抽出 overview-lite store

- 将概览级轻量数据从大 store 中拆出
- 保证概览页不再依赖完整详情数据链路

目标：

- 让概览页和常驻逻辑变轻

### 第三阶段：拆分 page-full stores

- 按处理器、显卡、主板、内存、存储逐步拆出页面 store
- 将详情页独有的派生数据、历史数据、重采样迁移到对应 store

目标：

- 建立清晰的模块边界
- 控制响应式传播范围

### 第四阶段：完善负缓存和链路权重

- 对 GPU、board telemetry、增强传感器链路做更精细的 TTL 和失败策略
- 区分“轻链路”和“重链路”的后台行为

目标：

- 进一步压低长期驻留开销

## 风险与取舍

### 需要避免的错误方向

- 单纯把所有轮询时间统一调大
  - 容易让用户感知“不实时”
  - 无法解决无意义采样问题
- 切页直接销毁组件
  - 虽然能进一步省资源
  - 但不符合本次“保留页面实例”的体验要求
- 概览页直接依赖完整详情数据
  - 会持续为重链路买单

### 可以接受的取舍

- 非当前页保留旧数据直到再次激活
- 后台状态下重链路暂停或大幅降频
- 重失败数据源进入较长时间负缓存

## 验证指标

后续实现完成后，至少验证以下指标：

- 插件前台静置时的平均 CPU 占用
- 插件后台静置时的平均 CPU 占用
- 插件运行一段时间后的内存占用是否稳定
- 概览页在默认“平衡”档下是否仍有足够的实时感
- 页面切换后是否能保留局部状态并快速回显
- GPU、主板、增强传感器链路在失败场景下是否减少重复尝试

## 最终建议

从长期看，应以“两级 store + 调度层”作为正式架构：

- `overview-lite` 承担轻量常驻刷新
- `page-full stores` 承担详情页独享的重数据
- `scheduler` 统一控制刷新档位、页面活跃态、窗口状态和链路权重

这样既能降低插件自身开销，也不会牺牲当前页面的使用体验，并且为后续继续扩展监控能力保留清晰空间。
