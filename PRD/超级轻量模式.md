# 硬件监控浮窗：超级轻量模式规划

## 1. 背景

当前插件已经存在一个硬件监控悬浮窗模式。

在办公场景下，桌面空间非常宝贵，现有悬浮窗虽然信息展示完整，但长期驻留桌面时存在感偏强。

因此新增一个 **超级轻量模式**，用于在不明显损失核心信息密度的前提下，将悬浮窗压缩到约 `200×200`，适合长期停留在桌面角落。

---

## 2. 模式关系

当前已有的悬浮窗模式保持不变，作为默认的标准模式继续存在。

本次新增的是一个独立的 **超级轻量模式**，不是替换现有悬浮窗，也不是重构标准模式。

模式关系：

* 标准模式：当前已有悬浮窗模式，本次不做额外规划
* 超级轻量模式：新增 `200×200` 高密度、低打扰悬浮窗

当前项目实现基线：

* 标准悬浮窗入口已存在：`plugin.json` 中的 `hardwareWatch`
* 标准悬浮窗页面已存在：`#watch` 渲染 `Watch` 组件
* 标准悬浮窗已经有 `overview / cpu / gpu` 三个内部视图
* 标准悬浮窗已经有 CPU / GPU / 内存采样、趋势 history、主 GPU 选择、运行时间、固定在最前等逻辑
* Windows / macOS 的硬件采集链路已经接入，包括 OpenHardwareMonitor、powermetrics helper、AppleSMC、macos-temperature-sensor 等

因此超级轻量模式应作为当前 `Watch` 悬浮窗的 **第二套渲染模式** 实现，复用现有采集链路、格式化函数、history、轮询策略和传感器增强状态，不新增一套独立硬件采集逻辑。

实现边界必须明确：

* `Watch/index.vue` 继续作为硬件监控浮窗的数据拥有者，负责 `window.services` 调用、轮询定时器、history 写入、主 GPU 选择、传感器增强结果消费
* `super-lite` 相关组件只接收 `Watch` 派生出的 readonly 展示数据、history 和交互回调
* `super-lite` 相关组件不得直接调用 `window.services.*`
* 第一阶段不得新增第二套硬件采集 composable、第二套轮询定时器或第二套 history
* 如果需要拆分数据整理逻辑，只允许拆成纯函数或纯展示 mapper，不允许在 mapper 内读取硬件服务

本次开发重点只关注：

1. 新增超级轻量模式独立入口
2. 补齐独立入口到 `Watch` 首屏 `super-lite` 的启动参数链路
3. 新增 `200×200` 超级轻量 UI
4. 支持标准模式与超级轻量模式互相切换
5. 持久化模式和固定状态

---

## 3. 超级轻量模式定位

超级轻量模式的定位是：

> 长期驻留、低打扰、高密度概览。

它不是把当前悬浮窗等比缩小，而是重新组织信息结构。

核心目标：

* 占用更少桌面空间
* 默认不显示低频信息
* 保留 CPU / GPU / 内存核心监控数据
* 视觉存在感明显低于当前悬浮窗
* 适合用户长时间挂在桌面边缘

---

## 4. 超级轻量模式核心原则

设计原则：

1. 默认只显示有实时价值的信息
2. 隐藏品牌、标题、导航等低频信息
3. 用高密度行级布局替代大卡片
4. 阅读态极简，操作态浮出
5. 默认低存在感，异常时增强提示
6. 信息不减少，只压缩表达方式

---

## 5. 默认布局结构

超级轻量模式默认尺寸：`200×200`

推荐结构：

```text
┌────────────────────┐
│ ● 良好        📌   │
│────────────────────│
│ CPU  9%   ▁▂▇▆ 52° │
│      1.1W     ▌    │
│────────────────────│
│ GPU 32%   ▁▃▂▆ 44° │
│      235mW    ▌▌   │
│────────────────────│
│ MEM 95%   ▁▁▇▃ 15G │
│      正常     ▌▌▌▌ │
│────────────────────│
│ ↻平衡  ⏱2天16:35   │
└────────────────────┘
```

整体结构：

```text
顶部状态栏：24px
CPU 行：48px
GPU 行：48px
内存行：48px
底部状态栏：20px
间距 / 分隔线 / 内边距：约 12px
```

## 6. 信息层级

### 一级信息

用户扫一眼最关心的内容：

* CPU 使用率
* GPU 使用率
* 内存使用率
* 当前整体状态

### 二级信息

保留但压缩展示：

* CPU 温度
* CPU 功耗
* GPU 温度
* GPU 功耗
* 内存已用
* 内存状态 / 压力

### 三级信息

以微型图形表达：

* Sparkline 趋势图
* 超细进度条
* 状态点

## 7. 顶部区域设计

超级轻量模式默认不显示 `HWInfoX`。

原因：

* 对用户没有实时价值
* 会增加窗口感
* 会占用宝贵空间
* 不适合长期驻留桌面

顶部区域只承担三个功能：

1. 拖拽区域
2. 整体状态提示
3. 悬浮窗控制入口

默认状态：

```text
● 良好        📌
```

说明：

* `● 良好` 表示当前整体状态
* `📌` 表示固定 / 取消固定
* 关闭、最小化、设置等低频操作默认隐藏，可在 hover 或右键菜单中出现

---

## 8. `概览 / CPU / GPU` 切换栏处理

超级轻量模式中，`概览 / CPU / GPU` 不常驻显示。

原因：

* 常驻切换栏会占用顶部空间
* 会让 `200×200` 版本看起来像缩小版应用窗口
* 违背“桌面仪表贴片”的设计目标

### 推荐方案：顶部状态栏复用

默认阅读态：

```text
● 良好        📌
```

鼠标 hover 顶部区域时：

```text
概览   CPU   GPU
```

鼠标离开 `1500ms` 后自动恢复：

```text
● 良好        📌
```

### 页面进入方式

除了顶部 hover 切换，也支持点击数据行进入详情：

* 点击 CPU 行：进入 CPU 详情
* 点击 GPU 行：进入 GPU 详情
* 点击内存行：进入内存详情

进入详情页后顶部显示：

```text
‹ CPU详情        ⋯
```

第一阶段详情页范围：

* CPU / GPU / 内存详情页只做 `200×200` 内的轻量信息列表
* 详情页复用现有 `Watch` 已有计算值和 history，但第一阶段不要求在详情页额外绘制趋势图
* 不在详情页里新增大型图表、完整传感器列表、诊断面板或配置入口
* CPU / GPU 详情可以参考当前标准悬浮窗已有详情数据，但视觉上需要重新压缩，不直接搬标准模式大卡片布局
* 详情页渲染不得触发新的硬件服务调用；进入详情页只允许改变 `Watch` 已有轮询 profile 或复用当前 profile

## 10. 底部状态栏

底部状态栏只保留最低限度信息。

推荐：

```text
↻平衡   ⏱2天16:35
```

也可以根据当前 `Watch` 轮询策略显示实际采样间隔：

```text
↻2.5s   ⏱2天16:35
```

可选追加：

```text
●良好
```

完整含义：

* `↻平衡` / `↻2.5s`：当前刷新档位或采样间隔
* `⏱2天16:35`：运行时间
* `●良好`：当前整体状态

不建议显示完整文字：

```text
刷新档位 平衡  运行时间 2天16:35:10  状态良好
```

原因是文字过长，会破坏 `200×200` 的轻量感。

## 11. 视觉风格

超级轻量模式保持当前悬浮窗的整体设计语言，但要明显降低存在感。

### 背景

* 深色半透明玻璃拟态
* 比当前悬浮窗透明度更高
* 不使用厚重实体背景
* 桌面长期驻留时不能显得突兀

### 边框

* 使用极细描边
* 默认弱发光
* hover 时轻微增强

### 分隔线

* 使用极淡分隔线
* 避免重卡片边框
* 行与行之间保持清晰但不强烈

### 颜色

沿用当前色彩语义：

* CPU：紫色
* GPU：青色
* 内存：绿色
* 正常状态：蓝绿色 / 青色
* 警告状态：黄色 / 橙色
* 危险状态：红色

## 12. 交互设计

### 默认态

* 不显示品牌名
* 不显示 `概览 / CPU / GPU`
* 不显示完整窗口控制按钮
* 只显示核心监控数据

### Hover 窗口

* 背景透明度略微提高
* 边框亮度略微增强
* 可显示隐藏操作入口

### Hover 顶部

顶部区域从状态栏切换为模式切换栏：

```text
概览   CPU   GPU
```

### Hover 数据行

对应行轻微高亮：

* 趋势图增强亮度
* 数值增强对比度
* 可点击进入详情页

### 双击（第二阶段）

第二阶段可支持双击窗口或数据区域时，临时显示完整字段名，持续约 1 秒。第一阶段不要求实现该交互。

默认：

```text
CPU  9%   52°  1.1W
```

双击后临时显示：

```text
CPU 使用率 9%   温度 52°   功耗 1.1W
```

这与“默认克制，主动查看时展开”的设计理念保持一致。

## 13. 详情页设计

超级轻量模式可以进入详情页，但详情页也应保持轻量。

第一阶段的详情页是“轻量详情”，用于满足行点击后的快速查看；复杂详情页放到第二阶段。

第一阶段轻量详情页必须保持固定、可测、低成本：

* CPU 详情：最多展示使用率、温度、功耗、频率、核心数
* GPU 详情：最多展示使用率、温度、功耗、显存占用
* 内存详情：最多展示使用率、已用、总量、压力 / 状态
* 所有字段都来自 `Watch` 已有 computed 或已有 service 结果
* 不显示完整传感器列表、不新增诊断面板、不新增配置入口
* 不新增详情页专属轮询定时器
* 若某字段缺失，展示 `--` 或已有的缺失原因短文案，不把缺失本身判为异常

### CPU 详情页示例

## 14. 异常状态

超级轻量模式不使用强打扰弹窗，优先通过局部视觉变化表达异常。

第一阶段状态判定使用固定阈值，避免实现和验收各自解释：

| 状态 | 触发条件 |
| --- | --- |
| 正常 | CPU / GPU 温度均低于 `85°C`，CPU / GPU 使用率均低于 `85%`，内存使用率低于 `90%`，macOS 内存压力不是 `warning` / `critical` |
| 注意 | CPU 或 GPU 温度达到 `85°C`；CPU 或 GPU 使用率达到 `85%`；内存使用率达到 `90%`；macOS 内存压力为 `warning` |
| 危险 | CPU 或 GPU 温度达到 `95°C`；CPU 或 GPU 使用率达到 `95%`；内存使用率达到 `95%`；macOS 内存压力为 `critical` |

说明：

* 同时命中多个状态时取最高严重程度
* 温度、功耗、GPU 或内存压力字段缺失时，不因为缺失本身进入注意 / 危险
* 字段缺失只影响对应辅助指标展示，可显示 `--` 或已有短文案
* Windows / 非 macOS 场景没有内存压力时，使用内存使用率判断

### 正常

```text
● 良好
```

### 注意

```text
● 注意
```

表现：

* 对应行颜色增强
* 第一阶段只要求静态颜色 / 状态点变化
* 底部状态变化

### 危险

```text
● 高温
● 高负载
● 内存紧张
```

表现：

* 对应行颜色切换为警告色
* 顶部状态点变色
* 可选轻微呼吸动效放到第二阶段
* 不主动弹出大面积提示

## 15. 入口设计

超级轻量模式作为新增模式，不替换当前悬浮窗模式。

### 插件独立入口

超级轻量模式必须有自己的 uTools 插件入口，用户可以不经过标准悬浮窗，直接从 `plugin.json` 入口打开超级轻量模式。

推荐新增 `features` 项：

```json
{
  "code": "hardwareWatchSuperLite",
  "explain": "超级轻量硬件监控",
  "cmds": [
    "超级轻量监控",
    "轻量监控",
    "硬件轻量监控"
  ]
}
```

入口行为：

1. 通过 `window.exports.hardwareWatchSuperLite` 进入
2. 直接创建硬件监控悬浮窗
3. 初始尺寸使用 `200×200`
4. 初始渲染模式为 `super-lite`
5. 不先展示标准悬浮窗
6. 不影响现有 `hardwareWatch` 标准悬浮窗入口

推荐实现映射：

```text
plugin.json features.code: hardwareWatchSuperLite
preload window.exports: hardwareWatchSuperLite
窗口预设: a_watch_super_lite
窗口页面: watch / #watch
启动参数: floatingMode=super-lite, entry=hardwareWatchSuperLite
初始渲染模式: super-lite
```

入口链路必须端到端补齐：

1. `plugin.json` 新增 `hardwareWatchSuperLite` feature
2. `utools/preload.js` 新增 `window.exports.hardwareWatchSuperLite`
3. preload 新增 `a_watch_super_lite` 窗口预设，尺寸为 `200×200`
4. `utools/services/window.js` 必须把 `a_watch_super_lite` 纳入 watch 窗口识别范围，与 `a_watch` / `watch` 一样加载 `watch` 页面，并保持透明、无边框、置顶、不可 resize 的 watch 窗口配置
5. 创建窗口时必须传递首屏启动参数，例如 `#watch?floatingMode=super-lite&entry=hardwareWatchSuperLite`，或等价的 per-window launch context
6. `App.vue` 必须能解析 `#watch` 后的 query 参数；解析页面名时不能把 `watch?floatingMode=super-lite` 误判为 `computer`
7. `App.vue` 在第一次渲染 `Watch` 前把初始 `floatingMode` 传给 `Watch`
8. `Watch` 用该初始值同步初始化 `standard / super-lite` 外层渲染模式，避免先渲染标准悬浮窗再切换

入口优先级：

* 从 `hardwareWatch` 进入时，首屏使用标准模式
* 从 `hardwareWatchSuperLite` 进入时，首屏使用超级轻量模式
* 用户在悬浮窗内手动切换模式后，再更新持久化设置
* 独立插件入口优先于上次持久化模式，保证入口语义稳定
* 只有后续存在“恢复上次模式”的通用入口时，才读取持久化的 `mode` 决定首屏模式

当前项目中已有标准悬浮窗入口：

```text
plugin.json features.code: hardwareWatch
preload window.exports: hardwareWatch
窗口预设: a_watch
```

因此超级轻量入口应作为新增入口并复用现有 `watch` 页面与采集链路，而不是新增一套硬件采集页面。

不得新增 `super-lite.html` 或独立的硬件采集页面。若构建脚本需要生成入口目录，也只能把它作为 `watch` 页面别名，不能让它承载另一套应用入口逻辑。

### 设置面板入口（第二阶段）

```text
显示模式
○ 标准模式
● 超级轻量模式

说明：
超级轻量模式会隐藏标题栏和导航栏，仅保留核心监控数据，适合长期驻留桌面。
```

### 浮窗右键菜单入口（第二阶段）

```text
右键菜单
├─ 切换到超级轻量模式
├─ 固定在桌面
├─ 透明度
├─ 更新频率
└─ 设置
```

### 当前悬浮窗快捷入口

在当前悬浮窗右上角增加一个轻量入口按钮。该按钮是标准模式第一阶段允许新增的唯一可见变化，标准模式原有布局、三种内部视图和采集行为保持不变。

按钮 tooltip：

```text
进入超级轻量模式
```

点击后切换为 `200×200` 超级轻量模式。

## 16. 模式切换行为

### 从当前悬浮窗切到超级轻量模式

行为：

1. 保留当前窗口位置
2. 将窗口尺寸切换为约 `200×200`
3. 隐藏品牌名和常驻切换栏
4. 展示超级轻量模式布局
5. 持久化当前模式为 `super-lite`

### 从插件独立入口进入超级轻量模式

行为：

1. 由 `plugin.json` 中的超级轻量入口触发
2. 通过 preload 中独立的 `window.exports.hardwareWatchSuperLite` 处理
3. 使用 `a_watch_super_lite` 预设创建 watch 窗口
4. 创建窗口参数中直接使用 `200×200`
5. 启动参数指定 `floatingMode=super-lite`，并在首次渲染 `Watch` 前生效
6. 首屏直接展示超级轻量模式布局
7. 不闪现标准悬浮窗布局
8. 进入后可将当前模式持久化为 `super-lite`

### 从超级轻量模式切回当前悬浮窗

行为：

1. 恢复当前悬浮窗模式
2. 恢复当前悬浮窗尺寸
3. 保留固定状态
4. 持久化当前模式为 `standard`

## 17. 状态持久化

需要持久化以下配置：

```ts
type FloatingMonitorMode = 'standard' | 'super-lite'

interface FloatingMonitorSettings {
  mode: FloatingMonitorMode
  pinned: boolean
  standardSize: {
    width: number
    height: number
  }
  superLiteSize: {
    width: number
    height: number
  }
  position?: {
    x: number
    y: number
  }
  opacity?: number
}
```

说明：

* `mode`、`pinned` 属于超级轻量模式第一阶段必须持久化的设置
* `standardSize` 默认使用当前标准悬浮窗预设尺寸
* `superLiteSize` 默认使用 `200×200`
* `position` 只有在窗口桥接支持读取 / 写入窗口 bounds 后才要求跨启动恢复
* `opacity` 属于第二阶段能力，第一阶段不提供自定义透明度时不得把它列为必须验收项
* 更新频率不在本结构中重复定义；超级轻量模式的底部刷新状态必须读取当前 `Watch` 轮询 profile，而不是新建刷新设置

第一阶段新增持久化 API 或存储键必须统一命名，推荐：

```ts
const FLOATING_MONITOR_SETTINGS_STORAGE_KEY = 'floatingMonitorSettings'

window.services.getFloatingMonitorSettings(): Promise<FloatingMonitorSettings>
window.services.updateFloatingMonitorSettings(
  patch: Partial<FloatingMonitorSettings>
): Promise<FloatingMonitorSettings>
```

如果第一阶段为了降低改动面暂不新增 `window.services` API，也必须在 PR / 实现说明中明确实际使用的 storage key，且只允许 `Watch` 读写，子组件不得自行读写 localStorage / dbStorage。

切换模式后应记住用户选择。

下次启动时：

* 如果通过标准悬浮窗入口进入，则首屏使用标准模式
* 如果通过超级轻量独立入口进入，则首屏使用超级轻量模式
* 如果后续存在“恢复上次模式”的通用入口，再读取 `mode`
* 同一窗口内从标准模式切换到超级轻量模式时，保留当前窗口位置
* 同一窗口内从超级轻量模式切回标准模式时，恢复标准悬浮窗尺寸
* 保留固定状态

### 窗口桥接要求

当前项目已有窗口尺寸调整能力：

```ts
window.services.resizeWindow(width, height)
```

第一阶段至少需要：

* 切到 `super-lite` 时调用 `resizeWindow(200, 200)`
* 切回 `standard` 时调用标准悬浮窗预设尺寸
* `resizeWindow` 当前只能调整内容尺寸，不负责读取或恢复坐标

如果第一阶段要求“跨启动恢复窗口位置”，则还需要补充窗口桥接：

```ts
window.services.getWindowBounds(): Promise<{
  x: number
  y: number
  width: number
  height: number
}>

window.services.setWindowBounds(bounds: {
  x?: number
  y?: number
  width?: number
  height?: number
}): void
```

如果暂不补窗口 bounds 桥接，则第一阶段验收只要求“模式切换时不主动改变窗口位置”，不要求重启后恢复窗口坐标。

## 19. 超级轻量模式组件职责

### SuperLiteMonitorView

职责：

* 渲染 `200×200` 根容器
* 管理当前页面：概览 / CPU详情 / GPU详情 / 内存详情
* 管理 hover 状态
* 管理顶部栏显示状态
* 管理行点击跳转
* 只消费 `Watch` 已有状态，不直接调用硬件采集服务
* 不读写 localStorage / dbStorage
* 不创建 interval / timeout 轮询；hover 恢复计时除外

### SuperLiteStatusBar

职责：

* 默认显示整体状态
* hover 时切换为模式切换栏
* 支持固定按钮
* 支持更多操作入口

### SuperLiteModeSwitcher

职责：

* hover 顶部时显示 `概览 / CPU / GPU`
* 支持切换到概览、CPU详情、GPU详情
* 不常驻显示
* 第一阶段 hover 离开后的恢复延迟固定为 `1500ms`，便于自动化测试

### SuperLiteMetricRow

职责：

* 渲染 CPU / GPU / 内存单行信息
* 接收颜色主题
* 接收主数值
* 接收趋势数据
* 接收辅助指标
* 支持 hover 和 click

### SuperLiteFooter

职责：

* 显示更新率
* 显示运行时间
* 显示可选状态
* 更新率显示必须来自当前 `Watch` poll profile，例如 overview 的 `WATCH_MODE_POLL_PROFILES.overview.fast = 2500` 应显示为 `↻2.5s`

## 20. 数据结构建议

超级轻量模式的数据应从现有 `Watch` 状态派生：

* CPU 使用率：复用 `cpuPercent`
* CPU 温度：复用 `cpuTempValue / cpuTemperatureDisplay`
* CPU 功耗：复用 `cpuPowerValue`
* GPU 使用率：复用 `gpuPercent`
* GPU 温度：复用 `gpuTempValue`
* GPU 功耗：复用 `gpuPowerValue`
* 内存使用率：复用 `memoryPercent`
* 内存已用：复用 `getDisplayMemoryUsedBytes(memoData)`
* 内存压力：复用 `memoryPressureLabel`
* 趋势图：复用 `history.cpu / history.gpu / history.memory`

数据所有权规则：

* `SuperLiteMetric` 只是展示结构，不是新的数据源
* 生成 `SuperLiteMetric[]` 的逻辑必须是纯计算，输入来自 `Watch` 已有 refs / computed / history
* 不允许在 `SuperLiteMetric` mapper 中调用 `window.services`、启动定时器或写入 history
* CPU / GPU / 内存详情页也必须消费同一组派生数据
* 自动化测试应能通过 mock `window.services` 调用次数确认切换 `standard / super-lite / detail` 不会新增第二套采集链路

组件内部可以整理为下列展示结构，但该结构不应成为新的采集源。

```ts
interface SuperLiteMetric {
  key: 'cpu' | 'gpu' | 'memory'
  label: string
  usage: number
  trend: number[]
  color: 'purple' | 'cyan' | 'green'
  primaryExtra?: string
  secondaryExtra?: string
  status?: 'normal' | 'warning' | 'danger'
}
```

示例：

```ts
const metrics: SuperLiteMetric[] = [
  {
    key: 'cpu',
    label: 'CPU',
    usage: 9,
    trend: [8, 12, 20, 9, 7],
    color: 'purple',
    primaryExtra: '52°',
    secondaryExtra: '1.1W',
    status: 'normal',
  },
  {
    key: 'gpu',
    label: 'GPU',
    usage: 32,
    trend: [20, 28, 18, 40, 32],
    color: 'cyan',
    primaryExtra: '44°',
    secondaryExtra: '235mW',
    status: 'normal',
  },
  {
    key: 'memory',
    label: 'MEM',
    usage: 95,
    trend: [80, 82, 95, 90, 95],
    color: 'green',
    primaryExtra: '15G',
    secondaryExtra: '正常',
    status: 'warning',
  },
]
```

## 22. 验收标准

超级轻量模式完成后，应满足以下标准。实现时建议在根节点和关键控件上提供稳定的 `data-testid` 或 `data-*` 状态属性，避免验收依赖易变 CSS 类名。

1. `plugin.json` 中新增 `hardwareWatchSuperLite` 独立入口
2. preload 新增 `window.exports.hardwareWatchSuperLite`
3. 新增 `a_watch_super_lite` 窗口预设，初始内容尺寸为 `200×200`
4. `a_watch_super_lite` 被窗口服务识别为 watch 窗口，并使用透明、无边框、置顶、不可 resize 的 watch 配置
5. 从 `hardwareWatch` 进入时首屏为标准模式
6. 从 `hardwareWatchSuperLite` 进入时首屏直接为 `super-lite`，首帧根状态可断言为 `data-floating-mode="super-lite"` 或等价状态
7. 独立入口优先于持久化 `mode`，不会先进入标准模式再切换
8. 当前已有标准悬浮窗的 `overview / cpu / gpu` 视图、尺寸预设和采集行为保持不变
9. `super-lite` 默认尺寸为 `200×200`
10. `super-lite` 默认不显示 `HWInfoX`
11. `super-lite` 默认不常驻显示 `概览 / CPU / GPU`
12. hover 顶部时显示 `概览 / CPU / GPU`
13. 鼠标离开顶部后 `1500ms` 自动恢复状态栏
14. 三行概览保留 CPU / GPU / 内存核心信息
15. 每组信息包含使用率、趋势、辅助指标
16. 点击 CPU / GPU / 内存行进入对应轻量详情页
17. 轻量详情页只展示固定字段列表，不出现完整传感器列表、诊断面板、配置入口或大型图表
18. 支持从轻量详情页返回概览
19. 支持从当前悬浮窗切换到超级轻量模式，并调用 `resizeWindow(200, 200)`
20. 支持从超级轻量模式切回当前悬浮窗，并调用标准悬浮窗预设尺寸
21. 模式和固定状态可持久化
22. 同一窗口内模式切换时不主动改变窗口位置
23. 底部刷新状态显示当前 `Watch` poll profile，例如 overview 显示 `↻2.5s`
24. 异常状态按第 14 节阈值触发，状态点和对应行颜色可断言
25. `super-lite` 复用当前 `Watch` 的硬件采集、history 和轮询逻辑；切换 `standard / super-lite / detail` 不重复创建采集链路

## 25. 最终产品表达

当前悬浮窗模式：

> 保持不变，继续作为默认完整悬浮窗使用。

超级轻量模式：

> 挂得久，占得少。

超级轻量模式不替代当前悬浮窗，而是为长期驻留桌面的场景新增一个更轻、更克制的显示方式。
